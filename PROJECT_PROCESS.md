# Financial Track 项目过程总结

## 项目目标

本项目用于个人消费订单数据的本地化采集与整理，目标是在不接入平台私有接口、不抓包、不绕过平台风控的前提下，把多个消费平台中的订单信息沉淀为可继续处理的本地原始数据。

当前阶段的重点是完成四个平台的原始数据获取：

- 京东：订单页面导出为 PDF。
- 淘宝：订单页面导出为 PDF。
- 拼多多：安卓订单页面截图，并通过本地 AI 识别整理为 JSON。
- 美团：安卓订单页面截图，并通过本地 AI 识别整理为 JSON。

截至当前阶段，四个平台的原始数据获取链路已经完成。京东、淘宝以 PDF 作为原始数据归档形式；拼多多、美团以截图作为视觉输入，并已通过本地 AI 生成结构化 JSON。

## 总体技术路线

项目采用“平台采集入口 + 基础能力模块 + 本地数据产物”的方式推进。

京东和淘宝的订单数据主要来自 Web 页面。由于页面本身适合打印归档，因此保留为 PDF，后续可以再接 OCR、PDF 解析或人工复核。

拼多多和美团的订单数据主要来自安卓 App 页面。项目通过 ADB 获取手机或模拟器截图，连续滚动订单列表，并在本地使用支持视觉输入的 llama.cpp OpenAI 兼容服务，将截图中的可见订单卡片整理为 JSON。

所有真实采集数据、日志、浏览器登录状态、本地模型路径和环境变量都保存在本机，不提交到版本库。

## 阶段一：项目基础结构

项目先搭建了基本工程结构：

- 根目录入口脚本承接独立工作流，例如 `jd_pdf_bot.py`、`taobao_pdf_bot.py`、`pdd_order_bot.py`、`meituan_order_bot.py`、`order_image_ai.py`。
- `order_capture/` 封装 ADB 截图能力。
- `src/localai/modules/` 放置本地 AI、配置、JSON 解析等可复用模块。
- `src/localai/flows/` 放置围绕具体任务的流程编排。
- `config.yaml` 和 `common.env` 分离通用配置与本机私有配置。
- `raw_data/` 和 `log/` 作为本地运行产物目录，并通过 `.gitignore` 排除。

这个结构保证了后续新增平台或新增处理流程时，可以优先复用已有的配置、日志、本地 AI 客户端和文件输出逻辑。

## 阶段二：京东 PDF 获取

京东链路由 `jd_pdf_bot.py` 实现。

实现方式：

- 使用 Playwright 启动 Chromium 持久化上下文。
- 浏览器用户数据目录保存在 `raw_data/jd_browser_profile`，用于复用登录状态。
- 根据 `config.yaml` 中的 `jd_order_pages` 配置生成年份和页码任务。
- 访问京东订单列表 URL，等待页面加载和必要的手工登录。
- 使用 Playwright 的 `page.pdf()` 直接渲染 A4 PDF。
- PDF 文件按年份、页码和时间戳命名，输出到环境变量配置的目录。

这一阶段解决了京东订单页面批量归档的问题，避免依赖系统打印对话框，也减少了鼠标键盘自动化的不确定性。

## 阶段三：淘宝 PDF 获取

淘宝链路由 `taobao_pdf_bot.py` 实现。

实现方式：

- 使用 `pyautogui` 操作已经打开的浏览器订单页。
- 启动后预留倒计时，人工把鼠标放到“下一页”按钮位置。
- 每轮点击下一页，等待页面加载。
- 触发 `Ctrl+P` 打开浏览器打印流程。
- 依赖浏览器默认打印目标为“另存为 PDF”。
- 自动输入按日期和页码生成的 PDF 文件名。
- 循环执行，直到用户按 `Ctrl+C` 或触发 pyautogui 安全停止。

这一阶段使用的是半自动方案，原因是淘宝页面和浏览器打印行为对登录状态、弹窗、页面布局更敏感。该方案保留了人工可控性，同时减少了重复翻页和保存文件的机械操作。

## 阶段四：拼多多和美团截图获取

拼多多、美团链路由 `pdd_order_bot.py`、`meituan_order_bot.py`、`android_order_workflow/android_order_bot.py` 和 `order_capture/capture_screen.py` 实现。

实现方式：

- 通过 ADB `exec-out screencap -p` 获取当前安卓屏幕截图。
- 支持显式指定设备序列号，也支持自动查找 PATH 中的 ADB 或 MuMu 自带 ADB。
- 平台配置中区分文件名前缀、输出目录和默认滑动参数。
- 支持三种采集方式：
  - `capture`：截取当前屏幕。
  - `capture-scroll`：按固定页数截图并滑动。
  - `capture-until-end`：截图、滑动并通过相邻截图相似度判断是否到达列表底部。
- 拼多多使用较短滑动距离，保留更多截图重叠，降低订单金额或关键字段被滑过的风险。
- 美团在遇到“显示更多历史订单”等需要人工干预的页面时，可以暂停，人工处理后继续。

默认截图产物保存在：

- `raw_data/pdd/`
- `raw_data/meituan/`

这一阶段完成了移动端订单页面的批量视觉原始数据采集。

## 阶段五：本地 AI 识别与 JSON 整理

拼多多和美团截图识别链路由 `order_image_ai.py` 和 `src/localai/flows/order_image_extract.py` 实现。

实现方式：

- 入口脚本根据平台解析默认截图目录。
- 支持识别最新一张截图，也支持 `--all` 批量识别目录内所有 PNG。
- 本地 AI 客户端读取 `config.yaml` 中的 `llamacpp` 配置。
- 如果开启自动启动，会在需要时启动本机 `llama-server`。
- 调用 OpenAI 兼容的 `/chat/completions` 接口，并以 data URL 方式传入图片。
- 提示词要求模型只输出 JSON，不输出 Markdown 或解释。
- JSON 解析模块会兼容模型输出中可能出现的代码块或额外文本。
- 每张截图单独保存一个 JSON 文件。
- 识别进度和结果摘要写入 `log/order_image_ai.log`。

默认 JSON 产物保存在：

- `raw_data/order_json/pdd/`
- `raw_data/order_json/meituan/`

当前 JSON 顶层结构包括：

- `platform`
- `source_image`
- `orders`
- `warnings`

每条订单记录包括商家、状态、标题、规格、数量、实付金额、原价、运费、下单时间、订单号、物流、可见操作按钮、是否截断、置信度和备注等字段。

## 已形成的关键经验

1. Web 平台优先保留 PDF 原件。PDF 更适合作为可复核的原始归档，也能减少对页面 DOM 或平台接口的依赖。
2. 移动端 App 优先用截图作为最小侵入采集方式。ADB 截图不需要修改 App，也不需要接入平台内部接口。
3. 连续截图必须保留一定重叠。订单卡片高度不固定，滑动距离过大容易漏掉金额、状态或订单标题。
4. 到底判断不能只依赖固定页数。通过截图相似度判断可以减少无效截图，同时保留手工继续的入口。
5. 本地 AI 输出必须有强约束。提示词、JSON 提取、warnings 字段和 raw output fallback 是保证批量处理可追踪的关键。
6. 真实路径和隐私数据必须和代码隔离。`common.env`、`raw_data/`、`log/` 和浏览器 profile 都不应进入版本库。

## 当前限制

当前阶段仍有一些边界需要后续处理：

- 京东和淘宝目前只完成 PDF 获取，尚未统一解析为订单级 JSON。
- 拼多多和美团截图识别依赖视觉模型能力，金额、订单号、截断订单仍需要抽样复核。
- 连续截图会产生重叠订单，后续需要订单级去重和跨截图合并。
- 不同平台字段命名和可见信息不完全一致，后续需要统一订单 schema。
- 本地 AI 运行依赖模型、mmproj、CUDA DLL 和 llama.cpp 版本，本机环境需要持续保持可复现说明。

## 下一阶段建议

后续可以按下面顺序推进：

1. 建立统一订单 JSON schema，覆盖 PDF 解析结果和截图 AI 识别结果。
2. 为拼多多、美团实现跨截图订单去重，优先使用订单号，其次使用平台、商家、标题、金额、时间等组合键。
3. 为京东、淘宝 PDF 增加解析流程，输出与移动端一致的 JSON。
4. 增加订单字段校验，例如金额格式、时间格式、必填字段、低置信度标记。
5. 增加人工复核清单，集中列出 `warnings`、`is_partial=true`、低置信度订单和解析失败文件。
6. 将四个平台 JSON 汇总为统一账本数据，再进入分类、统计或 Excel 写入流程。

## 阶段结论

本阶段已经完成多平台原始订单数据获取的核心闭环。京东和淘宝完成 PDF 归档；拼多多和美团完成截图采集，并已经通过本地 AI 转换为 JSON。项目现在具备继续推进“订单级清洗、去重、统一 schema、分类统计和报表输出”的基础。
